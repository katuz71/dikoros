# DIKOROS — API Findings

Архитектурные наблюдения по результатам аудита текущего API DIKOROS.
Источник: `openapi.json` (89 операций), `contract.md` (разметка USED/ADMIN/SERVICE/DEAD), snapshot прод-кода в `~/dikoros-audit/source-snapshot/`.
Дата: 5 мая 2026.

Каждый пункт: **что нашли → почему важно → что делать при миграции**.

---

## 1. Дублирование путей через префикс `/api` массовое и хаотичное

**Что.** Из 89 операций ~30 имеют дубликаты по схеме `/X` ↔ `/api/X` (товары, баннеры, категории, заказы, посты, чат, all-categories). При этом фронт RN и admin.html выбирают версии **рандомно**: мобайл зовёт `POST /api/auth`, но `GET /banners` (без префикса); админка зовёт `GET /api/orders`, но `POST /upload` (без префикса). Дополнительно есть третий стиль — `/api/admin/...` для двух ручек удаления юзеров.

**Почему важно.** Это удваивает поверхность атаки и тестирования. При миграции на Medusa придётся решить, какие пути сохранять для совместимости с мобайлом. Если оставить «как есть» — потащим хаос дальше; если унифицировать — сломаем мобайл до релиза новой версии RN-приложения.

**Что делать.** В Medusa использовать единый префикс `/store/...` для публичного API мобайла и `/admin/...` для админки (это конвенция Medusa). Старые пути дать как redirect-ы на нгинксе/Caddy в течение релизного периода (1–2 спринта), потом убрать. ADR про API-миграцию выделить отдельно.

## 2. Авторизация по `{phone}` в URL без токенов — критическая уязвимость

**Что.** Все приватные ручки идентифицируют юзера по номеру телефона прямо в пути: `GET /user/{phone}`, `GET /api/client/orders/{phone}`, `DELETE /api/client/orders/clear/{phone}`, `GET /api/user/reviews/{phone}`. Никаких bearer-токенов или session cookie не требуется. Любой, кто знает номер телефона клиента DIKOROS, может прочитать и удалить его заказы и отзывы.

**Почему важно.** Это нарушение GDPR/украинского закона о персональных данных. Утечка номеров (через любой источник — соцсети, прошлые заказы) превращается в полный доступ. Атаку можно автоматизировать перебором номеров украинских операторов (380XX...).

**Что делать.** Не реплицировать схему в Medusa. Medusa из коробки даёт JWT-аутентификацию через `/store/auth` и куку `_medusa_jwt`. При миграции мобайла переписать API-слой на токены. Записать в риски как **HIGH severity**, а в открытые вопросы — пункт «нужно ли уведомлять Юру и пользователей о ретроспективной утечке данных».

## 3. Прямые вызовы Новой Почты с фронта — утечка API-ключа

**Что.** Файл `app/checkout.tsx` мобильного приложения делает запросы напрямую к `https://api.novaposhta.ua/v2.0/json/`, минуя бэкенд. На бэке есть прокси-ручки (`/api/delivery/cities`, `/popular-cities`, `/warehouses`), но они используются непоследовательно.

**Почему важно.** Если ключ Новой Почты в open-source RN-приложении — он скомпрометирован. Любой может декомпилировать APK/IPA или прослушать трафик. У НП есть лимиты по запросам и тарификация — превышение оплачивает Юра.

**Что делать.** На неделе 1 проверить: есть ли реальный ключ в коде RN или используется публичный без авторизации. Если ключ — заменить, отозвать старый. В новой архитектуре все вызовы НП — через бэк-прокси, ключ только в env на сервере.

## 4. Контракт `Product` отсутствует в openapi

**Что.** В openapi есть 27 схем DTO (User, Order, Review, PromoCode и т.д.), но ни одной для товара — нет `Product`, `ProductCreate`, `ProductResponse`. При этом ручек товаров — 9 (с учётом дублей). FastAPI не сгенерил схемы автоматически — значит, эндпоинты возвращают `dict` или нетипизированный JSON.

**Почему важно.** Контракт продуктов придётся восстанавливать вручную из main.py и примеров реальных ответов. Это самый важный домен для каталога.

**Что делать.** На этапе миграции данных (Месяц 2) — снять реальные ответы `GET /products` и `GET /products/{id}` с прода через curl, сохранить как fixtures, по ним восстановить TypeScript-типы. В Medusa использовать стандартную модель `Product` без попыток скопировать «как было».

## 5. Категории — клиентская агрегация, не сущность

**Что.** Мобайл строит дерево категорий через `useMemo` из поля `product.category` (строка). Бэк-ручки `/all-categories` и `/api/categories` существуют, но мобайл их не зовёт — категории формируются на клиенте из массива товаров. На бэке категория как сущность, по факту, есть (CRUD в админке через `/categories`), но не используется фронтом.

**Почему важно.** При миграции на Medusa категории — это полноценные `ProductCategory` с иерархией, slug-ами, SEO-полями. Простая миграция «как есть» потеряет структуру: 100 товаров с `category: "Грибы"` дадут одну плоскую категорию без описания и SEO.

**Что делать.** На этапе миграции данных — сначала восстановить структуру категорий из админки (ручкой `GET /all-categories`), смапить на иерархию Medusa, затем привязать товары. Не доверять полю `product.category` как источнику истины.

## 6. Push-уведомления готовы на бэке, но не работают

**Что.** Эндпоинт `POST /api/user/push-token` существует в openapi (Save Push Token), но мобайл ни разу его не зовёт. Получается, бэк готов принимать токены и, вероятно, рассылать пуши, но клиенты их не регистрируют → никто пушей не получает.

**Почему важно.** Это упущенная возможность retention в текущем приложении. При миграции — либо реализуем правильно через Medusa Notifications + Expo Push, либо отказываемся осознанно.

**Что делать.** Спросить Юру на созвоне (пункт 6 Недели 1): нужны ли пуши вообще, и если да — какие сценарии (статус заказа? промо-акции?). Пока статус — открытый вопрос для клиента.

## 7. Ручка удаления аккаунта (`/delete-account`) не подключена в мобайле — блокер для App Store / Google Play

**Что.** Ручка `GET /delete-account` есть в openapi (статичная страница), но в мобайле нет UI-флоу удаления аккаунта. С 2022 года Apple и Google требуют функцию in-app account deletion для всех приложений, где есть регистрация.

**Почему важно.** При попытке выложить обновление мобильного приложения в стор оно будет отклонено. Это блокер релиза, а не «доработка позже».

**Что делать.** Записать в **риски** как HIGH severity. В план Месяца 3 (мобильное приложение под новый API) включить отдельной задачей UI удаления аккаунта + бэк-эндпоинт `DELETE /store/customers/me` (Medusa умеет это из коробки). Без этой задачи релиз заблокирован.

## 8. Опасная ручка `GET /api/clear_products`

**Что.** Эндпоинт `GET /api/clear_products` (summary: «Clear Products Db») очищает товары через **GET-запрос**. GET по HTTP-семантике должен быть idempotent и safe — то есть никогда не менять состояние. Любой префетчер (Chrome предзагрузка, кэш-пробы CDN, краулеры, антивирусы, link-preview-боты Telegram) может случайно уронить каталог.

**Почему важно.** Сейчас ручка не защищена ни авторизацией, ни CSRF-токеном, ни method-проверкой. Достаточно одного прохода Googlebot по случайно засветившемуся URL — и все товары исчезнут.

**Что делать.** Срочно проверить: есть ли реальная защита внутри (например, проверка origin, secret в query). Если нет — попросить Юру/разработчика прода добавить хотя бы IP-whitelist или header-secret до миграции. В Medusa такой ручки не будет вообще.

## 9. Расхождение `openapi.json` и реальной реализации

**Что.** Минимум две ручки **существуют в admin.html**, но **отсутствуют в openapi**: `POST /categories/{id}/banner` (single, не множественное) и `POST /api/import_xml` (импорт XML-фида). Для FastAPI это нетипично — schema генерится автоматически. Возможные причины: ручки реализованы через `@app.api_route(..., include_in_schema=False)`, либо это отдельные WSGI/ASGI-обработчики, либо мёртвые ветки в admin.html.

**Почему важно.** `openapi.json` нельзя считать единственным источником истины. При миграции импорт XML — критическая интеграционная точка с Хорошопом, её игнорировать нельзя.

**Что делать.** На этапе доступа к main.py (после получения паролей от Юры) — грепнуть по `import_xml` и `banner` в коде, разобраться, что это и как работает. Если подтвердится, что `import_xml` — рабочая ручка, добавить отдельным пунктом в план миграции данных Месяца 2.

## 10. `openapi.json` и `/docs` открыты публично

**Что.** Любой может зайти на `https://app.dikoros.ua/openapi.json` или `/docs` и получить полную схему API без авторизации. Я этим воспользовался для аудита, но злоумышленник получит то же самое.

**Почему важно.** Это упрощает разведку: сразу видно все ручки, включая опасные (`/api/clear_products`, batch-удаления, импорт XML). В сочетании с пунктом 2 (auth по phone) — почти готовый рецепт атаки.

**Что делать.** До миграции — закрыть `/docs` и `/redoc` базовой http-аутентификацией на уровне FastAPI или Caddy. `openapi.json` — туда же. После миграции на Medusa Admin API сам по себе доступен только с токеном.

## 11. Версионирование API — попытка, заброшенная на одной ручке

**Что.** В openapi есть единственная ручка с версией: `POST /api/v1/chat`. Все остальные 88 — без версии. Похоже на брошенный рефакторинг: кто-то начал унификацию, но не дошёл.

**Почему важно.** Само по себе не критично, но как сигнал — показывает, что у проекта нет архитектурной дисциплины. При миграции — возможность сделать сразу правильно.

**Что делать.** В Medusa использовать встроенные стабильные префиксы (`/store/v1/...`, `/admin/v1/...`) — Medusa 2.x именно так и делает. Не выдумывать собственное версионирование.

## 12. Файл `shop.db` (SQLite, 56 КБ) на проде рядом с FastAPI

**Что.** В `/opt/dikorosua` обнаружен `shop.db` — SQLite-файл на 56 КБ. В заявленном стеке проекта PostgreSQL, SQLite не упоминается. Назначение неизвестно: либо legacy от старой версии, либо локальный кэш, либо параллельное хранилище для какой-то фичи.

**Почему важно.** Если в нём есть данные, которые **не дублируются** в основной БД, при миграции они будут потеряны. Если это legacy — захламляет прод.

**Что делать.** На неделе 2 при доступе к серверу — открыть SQLite, посмотреть таблицы и записи. Решить: мигрировать содержимое или удалить. Записать в **открытые вопросы**.
